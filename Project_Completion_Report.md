## 1. 프로젝트 개요

### 1.1 프로젝트 명
**TripPrep (Intelligent RAG & TripPrep System)**

### 1.2 프로젝트 기간
2025.11.14 ~ 2025.12.05

### 1.3 참여 인원
서진주 (1명)

### 1.4 프로젝트 배경 및 목적
- **배경**: 해외여행이 익숙하지 않은 사용자는 여행 준비 과정에서 방대한 정보를 검색하고 정리하는 데 큰 어려움을 겪습니다. 또한, 여행 관련 문서를 일일이 읽고 분석하는 것은 시간 소모적인 작업입니다.
- **목적**: 
    1. **통합성**: 문서 분석(RAG)과 여행 정보 생성(TripPrep)을 하나의 플랫폼에서 제공하여 사용자의 편의성을 극대화합니다.
    2. **정확성**: RAG를 통해 문서 기반의 정확한 답변을 제공하여 환각(Hallucination)을 최소화하고, Tavily 검색을 통해 최신 여행 정보를 제공합니다.
    3. **효율성**: "정보 분석부터 여행 계획까지, AI 하나로 해결"이라는 컨셉으로, 사용자가 목적지와 키워드만 입력하면 맞춤형 가이드를 생성해줍니다.

## 2. 프로젝트 내용

### 2.1 개발 환경
| 구분 | 상세 내용 |
|------|----------|
| **OS** | Windows |
| **Language** | Python 3.x, TypeScript |
| **Framework** | **Backend**: FastAPI 0.117.1<br>**Frontend**: React 19.2.0, Vite 7.2.4 |
| **AI / LLM** | **Local**: llama.cpp (A.X-4.0-Light-Q4_K_M.gguf) - RAG 답변용<br>**Cloud**: Anthropic Claude 3.5 (Haiku/Sonnet) - TripPrep 리포트 작성용<br>**Embedding**: jhgan/ko-sroberta-multitask |
| **DB / Search** | **Vector DB**: FAISS (Facebook AI Similarity Search)<br>**Search API**: Tavily API |
| **Tools** | VS Code, Git, Notion API, pdfplumber |

### 2.2 주요 기능
1.  **RAG 챗봇 (Retrieval-Augmented Generation)**
    *   **다중 파일 처리**: PDF, TXT 파일을 동시에 여러 개 업로드 및 분석 가능.
    *   **정교한 전처리**: `pdfplumber`와 정규표현식을 활용하여 중복 문자 제거 및 문장 분리 최적화.
    *   **인터랙티브 검색**: 질문 시 Top-K 관련 문서를 실시간으로 보여주고, 사용자가 선택하여 답변 생성 가능.
    *   **투명한 출처**: 답변 생성에 사용된 문서의 페이지와 원문을 명확히 표시.

2.  **TripPrep (AI 여행 가이드 생성)**
    *   **맞춤형 리포트**: 목적지와 키워드(예: 맛집, 쇼핑) 입력 시 맞춤형 여행 준비 리포트 생성.
    *   **멀티 에이전트 시스템**:
        *   **Scout**: Tavily API로 실시간 웹 검색 (비자, 날씨, 최신 정보).
        *   **Architect**: 사용자 맞춤형 리포트 목차 설계.
        *   **Writer**: 수집된 정보를 바탕으로 고품질 마크다운 리포트 작성.
    *   **Notion 연동**: 생성된 리포트와 체크리스트를 사용자의 Notion 페이지로 자동 전송.

### 2.3 시스템 아키텍처
본 프로젝트는 **Hybrid AI Architecture**를 채택했습니다.
*   **Frontend (React)**: 사용자 입력을 받아 Backend API 호출 및 결과 렌더링 (Markdown).
*   **Backend (FastAPI)**:
    *   **RAG Pipeline**: 문서 업로드 -> 텍스트 추출 -> 청킹(Chunking) -> 임베딩(Embedding) -> FAISS 저장 -> 검색 -> Local LLM 답변 생성.
    *   **TripPrep Pipeline**: 사용자 요청 -> Multi-Agent (Scout/Architect/Writer) 협업 (Cloud LLM) -> 결과 생성 -> Notion API 전송.

### 2.4 데이터베이스 설계
전통적인 RDBMS 대신 **Vector Database (FAISS)**와 **Notion**을 데이터 저장소로 활용했습니다.
*   **Vector Store (FAISS)**:
    *   업로드된 문서의 텍스트 청크(Chunk)와 임베딩 벡터를 저장.
    *   고속 유사도 검색을 위해 인메모리 인덱스 사용.
*   **Notion**:
    *   최종 생성된 여행 리포트와 체크리스트를 영구 저장하고 관리하는 DB 역할 수행.

### 2.5 UI/UX 디자인
*   **반응형 디자인**: PC 및 모바일 환경 지원.
*   **다크 모드**: 눈의 피로를 줄이는 모던한 다크 테마 적용.
*   **직관적인 인터페이스**:
    *   좌측: 탭 메뉴 (RAG 챗봇 / TripPrep).
    *   중앙: 채팅 인터페이스 및 리포트 뷰어.
    *   우측: 문서 업로드 및 설정 패널.

## 3. 프로젝트 후기

### 3.1 프로젝트를 통해 얻은 가장 큰 보람 (400자)
이번 프로젝트의 가장 큰 성과는 **로컬 SLM(Small Language Model)과 클라우드 LLM을 결합한 하이브리드 아키텍처**를 성공적으로 구축한 것입니다. 비용 효율성을 위해 빈번한 질의응답(RAG)에는 로컬 모델을, 고품질의 추론과 작문이 필요한 리포트 생성(TripPrep)에는 클라우드 모델을 배치하여 성능과 비용의 균형을 맞췄습니다. 또한, **멀티 에이전트 시스템**을 구현하여 Scout, Architect, Writer가 협업하는 과정을 보며, 단일 프롬프트로는 불가능했던 복잡한 작업을 에이전트 간의 상호작용으로 해결할 수 있음을 확인했을 때 큰 보람을 느꼈습니다. 특히 생성된 결과물이 Notion으로 자동 연동되어 실제 여행 준비에 바로 사용할 수 있는 실용적인 서비스로 완성된 점이 매우 뿌듯합니다.

### 3.2 프로젝트를 진행하면서 가장 힘들었던 점 (400자)
가장 어려웠던 점은 **비정형 데이터(PDF)의 전처리** 과정이었습니다. 다양한 형태의 PDF 문서에서 텍스트를 추출할 때, 띄어쓰기가 없거나 중복 문자가 발생하는 등 노이즈가 많아 임베딩 품질이 저하되는 문제가 있었습니다. 이를 해결하기 위해 `pdfplumber`를 도입하고, 정규표현식을 활용한 커스텀 정제 로직(중복 문자 제거, 문장 분리 등)을 개발하는 데 많은 시간을 쏟았습니다. 또한, **멀티 에이전트의 속도 최적화**도 난관이었습니다. 각 에이전트가 순차적으로 작업을 수행하다 보니 전체 응답 시간이 길어지는 문제가 있었는데, 이를 개선하기 위해 프롬프트를 최적화하고 불필요한 검색을 줄이는 과정을 반복하며 튜닝하는 과정이 기술적으로 도전적이었습니다.

### 3.3 향후 프로젝트를 진행하게 될 후배 기수 훈련생들에게 전하고 싶은 말 (400자)
프로젝트를 시작할 때 **'어떤 문제를 해결할 것인가'**를 명확히 정의하는 것이 가장 중요하다고 생각합니다. 기술 스택을 정하기 전에, 사용자가 겪는 불편함이 무엇인지 깊이 고민해보세요. 저의 경우 '여행 준비의 막막함'과 '문서 분석의 번거로움'이라는 구체적인 문제에서 출발했기에 RAG와 TripPrep이라는 기능을 유기적으로 연결할 수 있었습니다. 또한, **최신 기술(LLM, Agent)을 두려워하지 말고 적극적으로 도입**해보시길 권합니다. 처음에는 어렵게 느껴질 수 있지만, 로컬 모델을 직접 돌려보고 API를 연동해보면서 얻는 경험은 실무에서 큰 자산이 될 것입니다. 마지막으로, 혼자 고민하기보다 멘토님이나 동료들과 자주 소통하며 피드백을 받는 것이 프로젝트의 완성도를 높이는 지름길입니다.
